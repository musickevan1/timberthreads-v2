# Plan 08-01: Performance Audit and Optimization

## Goal

Achieve Lighthouse 90+ on mobile Fast 3G, enforce the 50KB JS budget for client islands, confirm zero client-side JS for static sections, keep third-party embeds deferred, and fix the one known font-loading performance gap: self-host Inter instead of serving it from the npm package as a bundled CSS import.

## Context

The site is Astro 5 static-output with Tailwind v4. Current JS payload consists of:
- PhotoSwipe lightbox (loaded on gallery interaction, code-split)
- PricingCalculator Preact island (`client:load`)
- Nav scroll-spy + mobile menu (inline `<script>` in Nav.astro)
- Contact form handler (inline `<script>` in Contact.astro)
- Calendar iframe lazy-loader (inline `<script>` in Calendar.astro)
- Maps iframe lazy-loader (inline `<script>` in Map.astro)

The font is currently imported via `@fontsource-variable/inter` package in BaseLayout.astro — this bundles font files as CSS imports. Self-hosting via local files in `public/fonts/` eliminates any potential third-party request and gives us full control over font loading strategy (font-display: swap).

Both Google Calendar and Google Maps iframes already use IntersectionObserver lazy loading (rootMargin 200px). This satisfies PERF-05. Verify nothing regressed.

A11Y-01 (16px minimum / 18px preferred body text) belongs here because it directly affects performance tradeoffs — larger text can affect layout and render.

## Discovery Level

Level 0 — no new dependencies. All patterns established. Self-hosting Inter fonts is straightforward: copy from node_modules into public, update CSS.

<action>
**Step 1: Copy Inter Variable font files from node_modules to public/fonts/**

The `@fontsource-variable/inter` package installs font files at:
`node_modules/@fontsource-variable/inter/files/`

Run this to find the woff2 files needed:
```
ls node_modules/@fontsource-variable/inter/files/
```

Copy only the woff2 variable font files to `public/fonts/`. The key file is `inter-latin-wkst.woff2` (the full latin variable font). Also copy `inter-latin-ext-wkst.woff2` if present for extended Latin coverage. Do NOT copy the CSS files — we write our own.

Create `public/fonts/` directory and copy:
```bash
mkdir -p public/fonts
cp node_modules/@fontsource-variable/inter/files/inter-latin-wkst.woff2 public/fonts/
```

Verify the file copied and note the exact filename for the CSS step.

**Step 2: Replace the npm font import in BaseLayout.astro with a `<link>` preload**

Remove this line from `src/layouts/BaseLayout.astro`:
```
import '@fontsource-variable/inter';
```

Add in the `<head>` of BaseLayout.astro, BEFORE other stylesheets:
```html
<!-- Preload self-hosted Inter Variable font -->
<link
  rel="preload"
  href="/fonts/inter-latin-wkst.woff2"
  as="font"
  type="font/woff2"
  crossorigin
/>
```

**Step 3: Update global.css to use local font-face declaration**

Remove the Tailwind import for `@fontsource-variable/inter` if it appears in global.css (it should only be in BaseLayout.astro — confirm by checking global.css).

Add a `@font-face` declaration at the TOP of `src/styles/global.css`, ABOVE the `@import "tailwindcss"` line:

```css
@font-face {
  font-family: 'Inter Variable';
  font-style: normal;
  font-weight: 100 900;
  font-display: swap;
  src: url('/fonts/inter-latin-wkst.woff2') format('woff2');
  unicode-range: U+0000-00FF, U+0131, U+0152-0153, U+02BB-02BC, U+02C6, U+02DA, U+02DC, U+2000-206F, U+2074, U+20AC, U+2122, U+2191, U+2193, U+2212, U+2215, U+FEFF, U+FFFD;
}
```

The existing `--font-sans` theme variable already references `'Inter Variable'` — no change needed there.

**Step 4: Enforce 18px body text and audit small text utility classes (A11Y-01)**

In `src/styles/global.css`, the `body` rule currently sets `font-size: 1rem` (16px). Change it to `font-size: 1.125rem` (18px):

```css
body {
  font-family: var(--font-sans);
  font-size: 1.125rem; /* 18px — A11Y-01: preferred 18px body text */
  line-height: 1.6;
  color: var(--color-stone-800);
  background-color: white;
}
```

This may cause text overflow in small areas. After changing, run `npm run build` and visually check that no obvious layout breaks occur.

**Audit all `text-xs` and `text-sm` usage for WCAG compliance.** The locked decision is "strict 16px minimum everywhere (no exceptions)." Run:

```bash
grep -rn "text-xs\|text-sm" src/components/ src/pages/ src/layouts/
```

For every result, determine whether it is:
- **Decorative/non-content** (e.g., an icon label that is also visually represented, purely ornamental text): acceptable as-is if `aria-hidden="true"` is present on the element or its parent.
- **Content text** (readable by users for information): MUST be upgraded to at minimum `text-base` (16px).

Change all `text-sm` and `text-xs` instances on non-decorative content text to `text-base`. This includes navigation labels, form labels, captions, copyright notices, and any other text a user reads for information. The only permitted exceptions are elements with `aria-hidden="true"` where the information is conveyed another way.

### Task 3: Lighthouse score verification (PERF-01, PERF-03)

<files>
- No files modified — measurement only
</files>

<action>
**Step 1: Start the preview server**

```bash
npm run build && npm run preview &
```

Wait for the server to start (it will print the local URL, typically `http://localhost:4321`). Let it run in the background.

**Step 2: Run Lighthouse CLI against the preview server**

If `lighthouse` is not installed globally, install it once:

```bash
npx lighthouse --version || npm install -g lighthouse
```

Run Lighthouse in headless mode against the local preview server:

```bash
npx lighthouse http://localhost:4321 \
  --preset=perf \
  --throttling-method=simulate \
  --throttling.cpuSlowdownMultiplier=4 \
  --throttling.downloadThroughputKbps=1638 \
  --throttling.uploadThroughputKbps=768 \
  --output=json \
  --output-path=./lighthouse-report.json \
  --chrome-flags="--headless" \
  2>&1 | tail -5
```

The throttling values simulate Lighthouse's default "Mobile Slow 4G" (approximately Fast 3G). This matches the PERF-01 requirement ("Lighthouse 90+ on mobile Fast 3G").

**Step 3: Extract and evaluate scores**

After the run completes, extract the key scores:

```bash
node -e "
  const r = require('./lighthouse-report.json');
  const cats = r.categories;
  console.log('Performance:', Math.round(cats.performance.score * 100));
  console.log('Accessibility:', Math.round(cats.accessibility.score * 100));
  console.log('Best Practices:', Math.round(cats['best-practices'].score * 100));
  console.log('SEO:', Math.round(cats.seo.score * 100));
  const fcp = r.audits['first-contentful-paint'].numericValue;
  console.log('FCP (ms):', Math.round(fcp));
"
```

**Required passing thresholds (PERF-01, PERF-03):**
- Performance: **90 or higher**
- First Contentful Paint: **under 2000ms**
- Accessibility score is also visible here — target 90+

**Step 4: If Performance score is below 90 or FCP is above 2000ms**

Diagnose the top opportunities from the report:

```bash
node -e "
  const r = require('./lighthouse-report.json');
  const audits = r.audits;
  const failing = Object.values(audits)
    .filter(a => a.score !== null && a.score < 0.9 && a.details && a.details.type !== 'opportunity')
    .sort((a, b) => (a.score || 0) - (b.score || 0))
    .slice(0, 5);
  failing.forEach(a => console.log(a.score, a.id, a.title));
"
```

Address the top issues. Common fixes at this stage:
- Images too large: verify Astro's `<Image>` component is being used (not raw `<img>`), and that Cloudinary images have `width` and `height` props set
- Render-blocking resources: confirm the font preload `<link>` is before stylesheets in `<head>`
- Unused JS: verify PricingCalculator is using `client:visible` rather than `client:load` if it is below the fold

Iterate on fixes and re-run Lighthouse until the score is 90+ and FCP is under 2000ms.

**Step 5: Stop the preview server**

```bash
kill %1
rm -f lighthouse-report.json
```
